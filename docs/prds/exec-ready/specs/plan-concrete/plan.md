# Plan: plan-concrete

**Spec:** docs/prds/exec-ready/specs/plan-concrete/spec.md
**Charter:** .claude/skills/charter-*/SKILL.md (binding — each phase enumerates its honored NN-C/NN-P/CR entries)
**Status:** draft

## Overview

This piece adds a **plan concreteness contract** to the plan-authoring stage and a **concreteness floor** to the `qa-plan` gate, so a passing plan yields zero *unmarked* execute-time discoveries. It is a coordinated doc-as-code edit across five files: one new authoritative reference doc plus three citers plus the version-bearing descriptors.

**Track: Implement (non-TDD).** Front-matter declares `tdd: false`. All phases use the Implement track (`[Implement]` → `[Verify]` → optional `[Refactor]` → `[QA]`). Per the pi-011 testing doctrine these are prose-instruction files (SKILL/agent/template/reference) with **no executable unit tests** — so there is no `[Write-Tests]` block; each `[Verify]` step is a structural crit-presence read (`grep`/LLM-agent-step), and the adversarial gate is `qa-plan` + the end-of-piece Final Review. AC Coverage Matrix and Executable AC Binding are included for traceability even though non-TDD mode does not require them. `fast: false` — per-phase QA agents run inline.

**Dependency order.** Phase 1 creates the authoritative `reference/plan-concreteness.md`; Phases 2–4 each cite it (`plan/SKILL.md`, `qa-plan.md`, `templates/plan.md`); Phase 5 bumps the version across all four descriptors and runs the cross-phase citation-consistency verify. Phases 2–4 touch disjoint files but are kept serial deliberately (see each phase's `Why serial:` line) so per-phase Opus QA catches definitional/citation drift against the Phase-1 reference doc phase-by-phase.

**Dogfood.** This plan is itself an Implement-track doc-as-code plan and is subject to its own concreteness floor (every Change Specification Block names file + anchor + concrete content) and branch-enumeration-AC rule (every conditional branch in a deliverable maps to a numbered spec AC — see the AC Coverage Matrix). There are no genuine unknowns, so no `[SPIKE]` markers appear in this plan's prose.

### ADR-1: Reference-doc home for the concreteness contract
**Context:** The floor, the `[SPIKE]` marker, and the branch-AC rule must be defined once and consumed by three files (`plan/SKILL.md`, `qa-plan.md`, `templates/plan.md`). `[SPIKE]` has no definition anywhere in the plugin today (grep returns zero hits).
**Decision:** Create a new authoritative `plugins/spec-flow/reference/plan-concreteness.md` that defines all three concepts; the three consumers cite it and do not restate the definitions.
**Alternatives considered:** (1) Inline the definitions in `plan/SKILL.md` only — rejected: `qa-plan` needs the same definitions and would duplicate them, risking drift. (2) Extend `reference/spec-flow-doctrine.md` — rejected: that doc owns integration/TDD doctrine, not authoring-floor rules; a focused doc matches the `reference/research-artifact.md` single-source-of-truth precedent.
**Consequences:** Easier: one edit point for any future floor/marker change. Harder: introduces a fourth cross-file citation contract to keep aligned (mitigated by the Phase 5 cross-phase consistency verify). Honors the define-once convention.
**Charter alignment:** CR-005 (repo-root-relative paths), CR-008 (definitions are reference content; enforcement is split skill/agent).

### ADR-2: `[SPIKE: <unknown>]` syntax mirrors `[PENDING-DECISION: <area>]`
**Context:** The marker needs a syntax and a scan-scoping rule. The closest existing analog is `[PENDING-DECISION: <area>]`, an in-document unresolved-uncertainty marker the plan skill already scans for (skipping fenced code + HTML comments).
**Decision:** `[SPIKE: <free-text>]` — bracket + uppercase token + colon + free-text param. Scan scoping is copied verbatim from the Prerequisites `[PENDING-DECISION]` scan (skip fenced code + HTML comments; only raw prose markers count).
**Alternatives considered:** (1) A bespoke new syntax — rejected: needless divergence from an established marker family. (2) Reuse `[PENDING-DECISION]` itself — rejected: different lifecycle (PENDING-DECISION is a spec-stage marker cleared *before* plan; SPIKE is a plan-stage marker forward-handed to FR-005's resolver). Distinct from the `[RESEARCH-*]` family added by research-unify.
**Consequences:** Easier: reuses a proven scan idiom; `test-data-up` (FR-003) can reuse the same syntax. Harder: two visually similar marker families coexist (mitigated by distinct tokens + distinct lifecycles documented in the reference doc).
**Charter alignment:** NN-P-002 (every unknown is a visible marker, never a silent decision).

### ADR-3: Concreteness floor's primary gate is the concrete triple; the vague-verb list is a scoped illustrative signal
**Context:** A hard vague-verb blocklist grep would false-positive on legitimate occurrences ("the `[Implement]` block", "implementer agent") — the documented brittleness concern.
**Decision:** The floor's primary test is *presence of the triple* (target file + location/anchor + concrete content). Vague verbs are an illustrative signal scoped to deliverable/TARGET prose, not a standalone match.
**Alternatives considered:** (1) Hard vague-verb blocklist — rejected: brittle, false-positives. (2) Free-form "is this prose concrete enough" judgment — rejected: `qa-plan` cannot adjudicate subjective prose quality and has no codebase access.
**Consequences:** Easier: mechanically checkable from plan text alone (NN-C-008). Harder: the triple-presence check is slightly more elaborate than a single grep (acceptable).
**Charter alignment:** NN-C-003 (backward-compat — concrete pre-existing plans already carry the triple), NN-C-008 (evaluable from plan text alone).

### ADR-4: The version bump touches all four version-bearing descriptors, not three
**Context:** There are FOUR version files — `plugins/spec-flow/.claude-plugin/plugin.json`, `plugins/spec-flow/plugin.json` (root Copilot descriptor), `.claude-plugin/marketplace.json`, and `CHANGELOG.md` — all currently synced at 5.3.0. The spec's In-Scope + AC-6 name three ("`plugin.json`, `marketplace.json`, `CHANGELOG.md`"), but there are two `plugin.json` files.
**Decision:** Bump all four to 5.4.0; the Phase 5 `[Verify]` asserts all four equal. AC-6's named three are the verification minimum; the root Copilot `plugin.json` is included to avoid creating skew.
**Alternatives considered:** (1) Bump only the three named — rejected: would introduce a 5.3.0/5.4.0 skew between the two `plugin.json` files, violating NN-C-001 (the exact skew the later `sonnet-coord` piece is scoped to fix). (2) Defer the root descriptor to sonnet-coord — rejected: leaving a known skew across a release boundary contradicts NN-C-009.
**Consequences:** Easier: no introduced skew; sonnet-coord's "correct the skew" task is a no-op if no skew exists. Harder: one extra file in the bump series (trivial).
**Charter alignment:** NN-C-001 (plugin/marketplace sync), NN-C-009 (version bump on any `plugins/*` change).
