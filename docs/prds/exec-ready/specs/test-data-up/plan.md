# Plan: test-data-up

**Spec:** docs/prds/exec-ready/specs/test-data-up/spec.md
**Charter:** .claude/skills/charter-*/SKILL.md (binding — each phase enumerates its honored NN-C/NN-P/CR entries)

## Overview

This piece moves test-case design from execute to the plan. Every plan phase that **authors tests** — TDD-track `[TDD-Red]` phases and Non-TDD-mode `[Write-Tests]` phases — carries a per-case **`Test Data` block** (case ID + concrete input + expected oracle); the test-authoring agents (`tdd-red`, execute Step 2.7) transcribe it instead of designing live. The primary gate is a new `qa-plan` criterion #31; the agents' refuse-to-invent behavior is the backstop. It is a coordinated doc-as-code edit across **6 locations** — one new authoritative reference section (`reference/plan-concreteness.md` §5) plus 5 citers (`plan/SKILL.md`, `qa-plan.md`, `templates/plan.md`, `tdd-red.md`, `execute/SKILL.md` Step 2.7) — plus the 3 version descriptors + CHANGELOG.

**Track: Implement (non-TDD).** Front-matter declares `tdd: false`. All phases use the Implement track (`[Implement]` → `[Verify]` → optional `[Refactor]` → `[QA]`). Per the pi-011 testing doctrine these are prose-instruction files (SKILL/agent/template/reference) with **no executable unit tests** — so there is no `[Write-Tests]` block in this plan; each `[Verify]` step is a structural crit-presence read (`grep`/LLM-agent-step), and the adversarial gate is `qa-plan` + the end-of-piece Final Review. AC Coverage Matrix and Executable AC Binding are included for traceability even though non-TDD mode does not require them. `fast: false` — per-phase QA agents run inline.

**Dependency order.** Phase 1 extends the authoritative `reference/plan-concreteness.md` with §5; Phases 2-6 each cite §5 (`plan/SKILL.md`, `qa-plan.md`, `templates/plan.md`, `tdd-red.md`, `execute/SKILL.md`); Phase 7 bumps the version across all three descriptors + CHANGELOG and runs the cross-phase citation-consistency verify. Phases 2-6 touch disjoint files but are kept serial deliberately (see each phase's `Why serial:` line) so per-phase Opus QA catches definitional/citation drift against the Phase-1 §5 phase-by-phase.

**Dogfood.** This plan is itself an Implement-track doc-as-code plan and is subject to its own concreteness floor (every Change Specification Block names file + anchor + concrete content — qa-plan #28) and branch-enumeration-AC rule (every conditional branch in a deliverable maps to a numbered AC — qa-plan #30; see the AC Coverage Matrix). There are no genuine unknowns, so no `[SPIKE]` markers appear in this plan's prose. Note the new criterion #31 does NOT self-apply: this plan has no `[TDD-Red]`/`[Write-Tests]` phases.

### ADR-1: Extend the existing `plan-concreteness.md` with §5 rather than create a new reference doc
**Context:** The Test Data contract is the test-authoring analog of the per-phase concreteness floor and it reuses the `[SPIKE: <unknown>]` marker that `plan-concreteness.md` §2 already defines (which even carries a "Reuse by test-data-up (FR-003)" hook at §2 line 61). It must be defined once and consumed by 5 files.
**Decision:** Append a new `## 5. Test Data contract` section to the existing `plugins/spec-flow/reference/plan-concreteness.md`; the 5 consumers cite §5 and do not restate it. §5 cites §2 for the marker rather than redefining it.
**Alternatives considered:** (1) A new standalone `reference/test-data-contract.md` — rejected: fragments a single concreteness/test-authoring contract across two files and would have to re-point at §2's marker, risking marker-token drift. (2) Inline the contract in `plan/SKILL.md` only — rejected: `qa-plan`, `tdd-red`, and execute Step 2.7 all need the same definitions and would duplicate them.
**Consequences:** Easier: one contract doc, one marker family; the §2-reuse hook becomes a real cross-reference. Harder: `plan-concreteness.md`'s opening contract line (which enumerates citers) must grow to name the 2 new citers (mitigated — a single T in Phase 1).
**Charter alignment:** CR-005 (repo-root-relative paths), CR-008 (definitions are reference content; enforcement split across skill/agent), NN-P-002 (the per-case `[SPIKE]` is the visible-unknown mechanism §5 points at).

### ADR-2: Scope = both `[TDD-Red]` and Non-TDD-mode `[Write-Tests]` phases (any test-authoring phase)
**Context:** FR-003's text names "TDD phases" and `tdd-red`, but the same "design test data live" problem exists in Non-TDD-mode `[Write-Tests]` phases (authored by execute Step 2.7's generic dispatch). The sibling criterion #30 already treats Non-TDD-mode phases as in-scope.
**Decision:** The Test Data contract applies to any phase containing a `[TDD-Red]` **or** `[Write-Tests]` step. Pure `[Implement]` phases (no test step) author no tests and are out of scope. This added the 5th touch-point (execute Step 2.7) beyond the PRD's TDD-only framing — a deliberate human brainstorm decision (spec Q4).
**Alternatives considered:** (1) `[TDD-Red]`-only per FR-003's literal text — rejected by the operator: leaves the same gap in Non-TDD mode. (2) All phases — rejected: pure config/doc `[Implement]` phases write no tests, so a Test Data requirement there is ceremony.
**Consequences:** Easier: consistent with criterion #30's Non-TDD treatment; closes the gap on both test-authoring paths. Harder: a 5th edited file (execute Step 2.7) and qa-plan #31 must trigger on two markers.
**Charter alignment:** NN-P-005 (test design on the Opus plan stage; Sonnet transcription in both `tdd-red` and execute), NN-C-008 (criterion #31 evaluable from plan text alone for both markers).

### ADR-3: Backward-compat via an explicit absent-vs-incomplete split (`[TEST-DATA-ABSENT]` fallback vs `BLOCKED`)
**Context:** NN-C-003 requires the change to be additive — a plan authored before this contract has no Test Data block and must still execute. But a new-contract plan with an *incomplete* block is a real gap that must surface (NN-P-002).
**Decision:** Two distinct conditions, two distinct agent responses, stated explicitly in §5 and in both agents so they cannot be conflated: **block absent** → emit `[TEST-DATA-ABSENT: <reason>]` + fall back to today's design-from-assertions behavior (no block, no block); **block present but incomplete** (a named behavior with no covering case, or a case missing input/expected) → emit `BLOCKED` + route to plan amendment (Step 6c), write no partial test set. `qa-plan` #31 gates new plans, so absence at execute time means "legacy," letting the fallback be safe.
**Alternatives considered:** (1) Hard-require a block always, even when absent — rejected: breaks pre-5.5.0 plans, violates NN-C-003. (2) Invent the missing data — rejected: violates NN-P-002 (silent execute-time self-resolution). (3) Single response for both absent and incomplete — rejected: conflates legacy compatibility with a real new-contract gap.
**Consequences:** Easier: legacy plans run untouched; new gaps surface loudly. Harder: both agents must document two paths and the distinguishing condition (one extra branch each — covered by AC-5).
**Charter alignment:** NN-C-003 (additive), NN-P-002 (no silent self-resolution; gap → Step 6c).

### ADR-4: Per-case `[SPIKE]` reuses `plan-concreteness.md` §2 verbatim — no new marker token
**Context:** An unpredictable expected outcome needs a marker. `[SPIKE: <unknown>]` (§2) already exists with syntax + scan-scoping + finalize-gate behavior; §2 explicitly anticipates this reuse.
**Decision:** An unpredictable case's expected-outcome position carries `[SPIKE: <unknown>]` (per-case granularity); predictable cases keep concrete data. §5 cites §2 for the grammar and introduces no `[SPIKE-DATA]` or other variant. The existing plan-finalize spike-scan (shipped by plan-concrete) is the interim guard — this piece edits no finalize logic.
**Alternatives considered:** (1) A bespoke `[SPIKE-DATA: …]` token — rejected: needless divergence + marker-token drift across the now-5 citers. (2) Per-phase `[SPIKE]` (whole phase deferred) — rejected: loses the concrete predictable cases in a mixed phase; per-case is strictly more expressive.
**Consequences:** Easier: one marker family, one finalize gate, zero new finalize logic. Harder: none material — the §2 scan-scoping already covers fenced/comment exclusion.
**Charter alignment:** NN-P-002 (every unknown a visible marker), NN-C-003 (no change to existing marker/finalize behavior — additive).
